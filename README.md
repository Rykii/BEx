# BEx — FICC业务探查智能体系统 v2.0

[BEx](https://github.com/Rykii/BEx) 是 FICC（固定收益、货币、商品）业务探查系统的 **主控智能体（Orchestrator）+ Backlog Manager**，采用五级闭环架构，为 FinTech 资产管理产品经理提供官方规则探查与非置信源交叉验证能力。

## 🎯 核心概念

| 概念 | 说明 |
|------|------|
| **FICC** | 固定收益(Fixed Income)、货币(Currency)、商品(Commodities) |
| **业务探查** | 对特定资产类型的监管规则、交易制度、清算规则进行全面调研 |
| **五级闭环** | Recon → Archivist → Analyst → Verifier → Backlog Manager |
| **Gap 闭环** | Analyst 输出的待核实事项自动触发定向探查，最多 3 轮 |
| **非置信源校验** | 研报/会议纪要等非官方文件，经 Verifier 交叉验证后才能入库 |
| **知识库** | 基于官方文件生成的标准化 Markdown 文档集合 |

## 🏗️ 五级闭环架构

```
用户请求 / 用户上传文件
    │
    ▼
BEx（调度 + Backlog Manager）
    │
    ├── 路径 A：官方规则探查
    │       Recon → Archivist → Analyst ──→ 产出含【待核实】清单的文档
    │                                        │
    │                                        ▼
    │                               Backlog Manager 检测未闭合 Gap
    │                                        │
    │                               是 ──→ 触发 Recon（定向搜索模式）
    │                               否 ──→ 结束，文档定稿
    │
    └── 路径 B：非置信源处理
            Verifier 提取断言 ──→ 逐条交叉验证 ──→ 输出【印证/证伪/存疑】
                            │
                            ▼
                    官方印证的断言 ──→ 回写知识库（带双源引用）
                    官方证伪的断言 ──→ 生成勘误表，禁止入库
                    无法证实的断言 ──→ 保留但标注置信度 Low
```

## 🔁 迭代探查（Gap 闭环）

### 闭合标准

| 轮次 | 行为 | 输出要求 |
|------|------|---------|
| **Round 1** | 广撒网，建立基线文档 | 产出含 Gap 清单的 v1 文档 |
| **Round 2** | 定向搜索，聚焦 P0/P1 Gap | 更新文档，闭合已解决 Gap |
| **Round 3** | 最后一轮，同义词/跨市场类比 | 仍未闭合的强制标记为【监管空白】 |
| **> Round 3** | 停止自动探查，人工介入 | 输出 `STALE_REPORT`，附未闭合 Gap 清单 |

### Gap 优先级

| 优先级 | 含义 | 判定标准 |
|--------|------|---------|
| **P0** | 阻塞产品设计 | 无法确定则核心业务流程不可设计 |
| **P1** | 重要 | 影响系统功能模块边界或风控逻辑 |
| **P2** | 优化 | 影响非核心功能或体验优化 |

## ✅ 非置信源交叉验证

### 验证结果处理

| 结果 | 含义 | 处理方式 |
|------|------|---------|
| **CORROBORATED** | 官方文件印证 | 回写知识库，引用格式 `【{报告} 经 {官方文件} 印证】` |
| **CONTRADICTED** | 官方文件证伪 | 生成勘误表 → `_meta/corrections/`，**禁止 Analyst 采信** |
| **UNVERIFIABLE** | 官方未提及 | 写入 `probe_backlog.json` 作为新 Gap，触发定向探查 |

### 人工介入点

| 触发条件 | 行为 |
|---------|------|
| 达到 max_rounds 仍有 P0/P1 Gap | 输出 `STALE_REPORT`，询问用户决策 |
| Verifier 发现 CONTRADICTED 且涉及核心业务 | 暂停，推送勘误摘要 |
| 用户要求"直接基于未验证报告出文档" | **拒绝**，提示先完成交叉验证 |
| 两个官方文件对同一规则矛盾 | 暂停，对比展示矛盾条款 |

## 📖 使用方法

### 路径 A：官方规则探查

```
我需要了解【外汇NDF】的【监管框架和交易规则】
```

BEx 自动执行：Recon（标准）→ Archivist → Analyst → Backlog Manager（迭代至闭合）

### 路径 B：非置信源校验

```
我上传了一份《碳金融市场业务解析报告.pdf》，请帮我核实后补充到探查文档
```

BEx 自动执行：Verifier（断言提取→交叉验证）→ 核验结果处理 → Analyst（更新文档）

## ✨ 关键约束

| 约束 | 要求 |
|------|------|
| **源追溯** | 所有内容溯源到官方域名，禁止第三方整理版本作为主源 |
| **版本优先** | 2024-2026 年有效版本优先；冲突时以最新未废止版本为准 |
| **Gap 闭环** | 每轮迭代后更新 `probe_backlog.json` 状态 |
| **非置信源先验证** | 用户上传非官方文件时，必须先经 Verifier 验证 |
| **禁止编造** | 未找到明确依据时标注【待核实】，禁止用"一般惯例"填补 |
| **客观性** | 不补充个人解释，保持原始监管语义 |
